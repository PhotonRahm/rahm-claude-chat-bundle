## Core operating principles

- **Evidence-gated scaling.** Position size increases and parameter changes require proof windows. 50 resolved trades minimum for Kalshi parameter changes. Statistical significance (Wilson 95% CI lower bound above break-even) before any cap lift.
- **Fix at the data layer, not the presentation layer.** Bugs go away at the source, not in display logic.
- **Permanent fixes over one-time patches.** Recurring issues get durable code fixes.
- **Surgical decisions over aggregate blocks.** Block specific (asset, side) combinations rather than entire assets when data supports it.
- **Asymmetry traps are the dominant structural risk.** A high win rate (88%+) with negative P&L means avg loss size dwarfs avg win size. This pattern has appeared in: Kalshi Longshot Fading (88% WR, -$205), ETH-YES MR (89% WR, -$276), KXETHD MR (91% WR, -$180), IBKR Economics shadow post-FRED-correction (95% WR, -$97). Always check break-even WR vs observed WR, not just observed WR alone.
- **Gap = bug.** Reconciliation drift is never "transient" without proof. Identify the source.

## Hard rules — do not violate without explicit user approval

- **No portfolio caps.** User has explicitly stated this. Do not propose portfolio-level exposure limits. Per-trade caps and circuit breakers are different and acceptable.
- **No flat-qty defaults masquerading as Kelly.** If a sizing path uses Kelly, log it as Kelly. If it uses a flat default, log it as flat. Mismatched logging is a recurring bug class.
- **No silent deploys.** Every change gets a git commit with a descriptive message and gets pushed to origin.
- **No batching reports.** Eric does not want consolidated multi-item reports. One item at a time, report as completed before moving to next.

## Operational patterns

- Strategy independence: each strategy has its own filter stack. MR_BLOCKED_PATTERNS only affects MR. Moderate Favorites uses MODERATE_FAVORITES_ALLOWED_ASSETS. They do not share blocklists.
- Sizing source-of-truth pattern: pre-order sizing, pre-exec cap, post-fill verification, emergency auto-sell, and auto-sell-excess must all use the same cap function. Mismatches across these locations is the recurring "size in one place, cap in another" bug class. Found and fixed multiple times — including 2026-05-01 on Kalshi `check_oversized` (legacy `_position_cap_for_ticker` fallback was warning at cap=100 against MF qty=150 and MR Kelly-sized qty=102). When you encounter it again, fix all locations in one commit.
- Source-of-truth assertions must use AST inspection, not substring search. Substring search passes on a comment, docstring, or unreachable branch that mentions the symbol; AST inspection requires the symbol appear as an actual `ast.Call` node. The Gemini and Kalshi DS startup assertions both use this pattern (2026-05-01 hardening) — apply the same pattern to any new sizing/cap source-of-truth assertion.
- Cooperative shutdown requires bounded SDK timeouts. The `kalshi_python` SDK creates its urllib3 PoolManager with no default timeout; an in-flight SDK call will block past SIGTERM and force systemd to escalate to SIGKILL (happened once on 2026-05-01). Any new SDK integration must verify default timeout behavior; if absent, monkey-patch the rest_client's `request` to inject a sensible `_request_timeout=(connect, read)` default.
- Scaling env vars are mandatory, not optional. `MAX_EXPOSURE`, `MAX_POSITIONS`, `QTY_PER_POSITION` (IBKR) and equivalent caps elsewhere should fail-loud at startup if missing. A silent fallback to old defaults would let a misconfigured deploy trade at quarter-scale with no operator alarm. Use a `_require_env_scaling` helper that raises `SystemExit` with a `FATAL` message rather than `os.getenv("X", "old_default")`.
- Shadow observation freshness must mean field freshness, not just timestamp freshness. If a live strategy consumes shadow rows, use insert-per-observation or update every execution-relevant field on each upsert. Never refresh only a timestamp/count while leaving prices, depth, edge, or executable flags stale; this caused Gemini DS to report fresh qty50-executable opportunities from old market snapshots on 2026-05-01.
- Data freshness monitoring is mandatory for every table or feed that can affect live decisions or shadow promotion decisions. Log aggregation catches explicit errors; it does not catch a cron that exits cleanly while writing no rows, a disabled feed, or a stale cache. Any new decision feed must be registered in a freshness checker with a concrete max-age threshold and must surface stale/missing status in daily reporting. The 2026-05-02 NWS incident left `shadow_nws_frontrun` and `shadow_nws_kalshi` stale for seven days because the NWS shadow was deprecated by flag while downstream consumers still read its tables.
- Contract-universe enumeration must distinguish current-run coverage from historical cache size. A page cap, stale cached rows, or conflating cumulative table rows with active rows can make strategy coverage look materially larger or smaller than reality. Any enumeration used for strategy decisions must log page count, cursor exhaustion status, API throttling, and whether counts are "seen this run" or "stored historically."
- Resolution data must come from authoritative settlement state, not inferred sibling outcomes or broad event status. For multi-leg/categorical markets, do not mark every non-winning active leg as resolved just because one leg has a winner; require individual contract settlement metadata, explicit resolution side, exchange settlement rows, or an authoritative external actual matched to that exact contract/leg. This prevents fresh-looking but false resolved P&L in shadow promotion analysis.
- Event-state correctness is not market-EV correctness. A detector can be perfect on outcomes while still not be live-deployable if it lacks contemporaneous quote/depth data from the venue. Promotion analysis for time-sensitive DS/event strategies must include forward quote capture at the actual detection timestamp, linked resolution, spread/depth statistics, and a live-execution friction model.
- New shadow strategies must start with insert-per-observation storage, a single source-of-truth helper for fee/spread/slippage/threshold calculations, strategy independence from existing live blocklists/circuit breakers, and disabled live gates. Standalone shadow services must handle SIGTERM cooperatively and log clean shutdown.
- Synthetic tests must demonstrate end-to-end data flow, not just isolated component math. A test that exercises the function in isolation can pass while the deployed pipeline is structurally non-functional. The 2026-05-01 Tier 1 deploy passed 5/5 synthetic component tests but the trade-resolution → snapshot UPDATE linkage was missing entirely; the audit caught this only by inspecting the deployed DB. New shadow components must include at least one end-to-end test that walks insert → outcome update → analyzer/aggregator output. The dedup, classify, and outcome-update tests in `scripts/test_tier1_shadow.py` are the current reference pattern.
- Linked shadow tables must ship with linkage and resolution code in the same deploy. If a component logs signals that later resolve, include the resolver path and an end-to-end test that inserts, resolves, and proves reports can read the result.
- Linked-table components must be wired at deploy time, not deferred. A schema with foreign-key columns (e.g. `trade_id`, `outcome_resolution`) and no code populating those columns is structurally non-functional. The 2026-05-01 Tier 1 Adverse-Selection Filter shipped with `trade_id` 0% populated and zero `UPDATE orderbook_snapshots` callsites; the analyzer returned zero feature rows on every run. The 2026-05-01 Tier 2 deploy repeated the same pattern across all three components (CVD/MRN/VIBP) — only `resolve_synthetic` (a test helper) updated `resolved=1`, no production resolver existed. Both incidents required immediate follow-up commits with linkage and resolver code. When designing components with linked tables, stage the linkage code (insert path passes the FK; resolution path UPDATEs the linked row) into the same deploy as the schema, never as a follow-up.
- Helper functions named "synthetic" / "test_only" must not be the only path that populates production columns. If `resolve_synthetic` is the only function in the codebase that updates `resolved=1` on a production table, the component is structurally non-functional regardless of how comprehensive the unit tests are. The 2026-05-01 Tier 2 deploy had a pristine `test_end_to_end_*_resolution` test that exercised `resolve_synthetic`; the test was correct but the only thing in production calling it was the test runner. Production resolvers must exist with a non-test name and be invoked by cron, scheduled service, or settlement code.
- Methodology helpers must source from authoritative data, not heuristics. Functions that produce `predicted_side`, `predicted_probability`, or `model_probability` fields on shadow rows must derive these from contract structure, market data, or proper models — not from hardcoded values or simple time-decay heuristics. Heuristic-based observations should be explicitly flagged with `methodology_class='heuristic_observation'` and `actionable=0` so they don't flow to deployment recommendations. The 2026-05-02 audit found `_market_observation_from_active` hardcoded `side="YES"` and called a flat time-decay heuristic for model probability across 9 Gemini non-crypto categories and the Kalshi other-bucket; the resulting Gemini sports 5.8% WR was not a market inefficiency, it was the methodology. Audit-time grep for hardcoded `side="YES"` in shadow detectors and flat probability returns should find none in production code paths populating model fields. Authoritative path → favorite-tracking path (price-derived side) → heuristic path (provenance only, actionable=0) is the routing order.
- Promotion code must reproduce audited shadow rows before going live. Any live strategy promoted from shadow data needs an equivalence harness that feeds sampled audited shadow inputs into the live implementation and proves side, market price, model probability, edge, and actionability match. Do not deploy from prose descriptions when source code and description disagree.
- Timing analysis must use source observation timestamps, not derived `detected_at` or scan-cycle timestamps. Shadow normalization, scanner backfills, and live promotion analyses often rewrite `detected_at`; for original timing, use the source table timestamp/source row. This avoids stale-cycle and wrong-window deploys.
- Hardened-strategy parameter pattern. For strategies whose parameters are derived from audit findings, calibration analysis, or other process where the value selection cannot be casually reconsidered, hardcode the parameters in source code and reserve environment variables for enable/disable + kill switch path only. This is stronger defense-in-depth than full-env-config: configuration changes require code commits and review, accidental .env modifications cannot loosen safety parameters, and audit-derived values are anchored to the analysis that established them. Hardcode: cell/series allowlists derived from per-cell statistical robustness analysis, edge thresholds derived from calibration findings (e.g. "<40c is negative-EV"), CB limits derived from risk capacity analysis, position size caps derived from capital allocation decisions. Keep in env: enable/disable flag, kill switch file path. Reference implementation: IBKR Weather DS (`scan_trade_loop.py:225-242`) — env controls only `DETERMINISTIC_SETTLEMENT_WEATHER_ENABLED` + `_KILL_SWITCH`; QTY/MAX_OPEN/PER_TRADE_MAX/DAILY_CB/LIFETIME_CB/EDGE_THRESHOLD/ALLOWED_CELLS are hardcoded constants. Don't apply this pattern to strategies under active parameter exploration where rapid iteration is needed (e.g. during initial pilot evaluation).
- Principle 20: Hardened-strategy env vars must reach the running process. The pattern above is undermined if the bot starts but the env var never propagates from `.env` to the process. The 2026-05-02 IBKR Weather DS deploy had `DETERMINISTIC_SETTLEMENT_WEATHER_ENABLED=true` in `.env` but the systemd unit was missing `EnvironmentFile=`, so `os.getenv(..., "false")` at module import resolved to false and `place_ds_weather_trade` silently returned at the first-line gate. Verify after every hardened-strategy deploy: `tr '\0' '\n' < /proc/$(systemctl --user show -p MainPID <unit> --value)/environ | grep <VAR>` must show the variable. Bot run scripts that source `.env` themselves (e.g. `set -a; source .env; set +a`) are an alternative; pick one and stick with it. This is BUG_CLASSES #16: file-system config validation is not enough.
- Principle 21: Cache aggressively-repeated read endpoints; never cache write/orderbook/auth-ed endpoints. When the same read endpoint is queried multiple times per cycle by independent strategies (e.g. Kalshi `/markets` queried by MR, MF, DS, Index DS, FX DS for the same series), an in-process TTL cache cuts upstream rate-limit pressure without changing decision freshness materially. Pattern: opt-in cache (caller passes `cache_ttl > 0`) so orderbook/auth/write paths are never affected; cache only successful 200 responses, never error/throttle responses; thread the TTL through the public method as `cache_ttl_seconds` so hot paths can override (e.g. final-window DS uses 2s); add an AST source-of-truth assertion at startup so future refactors cannot drop cache routing silently. If cache alone does not keep 429s below target, add TTE-aware scan throttling before loosening strategy parameters. The 2026-05-02 Kalshi REST cache deploy turned 1,033 429s/14h into a controlled retry path with exponential+jitter backoff capped at 16s; same architecture applies anywhere a polled discovery endpoint feeds multiple consumers.
- Principle 22: Verification deferrals are tracked, not forgotten. If a verification cannot be performed at deploy time because it depends on a time-based event such as a settlement, publication, or measurement window, document the deferred verification with the trigger condition, due time, and owner. The trigger entering the past must surface as DUE_NOW in reports. Current source of truth: `~/operations-knowledge/VERIFICATION_DEFERRED.md`, mirrored into workspace reports through `scripts/verification_deferred.py`.
- Principle 23: Multi-section reports need a single data snapshot boundary. If a report embeds several subreports that query fast-moving bot DBs, snapshot the underlying SQLite DBs at report start or pass a common `as_of` through every query path. Otherwise the report can contradict itself without any real reconciliation bug. The 2026-05-03 `full_picture.py` fix snapshots Gemini/Kalshi/IBKR DBs and passes `RAHM_REPORT_*_DB` paths to embedded reports.
- Principle 24: Large operational SQLite integrity checks require a maintenance window. Once a shared DB is large enough that `PRAGMA integrity_check` competes with live writers, stop the shadow/non-live writers, verify `lsof` is clear, run `quick_check` and `integrity_check` offline, then restart services and confirm new inserts. For DS expansion, schedule partitioning/archive work once the DB exceeds 50GB or the offline integrity window exceeds 30 minutes.
- Principle 25: Position state must reconcile against platform settlement state on a defined cadence. For every live strategy that holds contracts to settlement, document expected settlement time, resolver cadence, and maximum acceptable hold time after settlement. A local open position past that threshold is a drift candidate; if the platform shows the contract resolved while the local DB is still open, treat it as a resolver bug. The detector is `~/.openclaw/workspace/scripts/detect_stale_positions.py` and it is read-only by design.
- Principle 26: External vendor failures get automated retry with backoff and non-spam escalation. Transient vendor failures such as IBKR Flex error 1001 should persist retry state, back off, and escalate once reporting freshness crosses the threshold. Credential/auth failures should not be hammered; mark them for human action. Retry reports must surface the raw vendor error code/message in addition to the local classification, so operator diagnosis is tied to the exact upstream failure. Once a retryable vendor failure persists beyond the normal freshness SLA, run targeted read-only probes that distinguish credential, query/config, endpoint, date-window, result-size, endpoint deprecation, and vendor-availability failure modes; passive retry alone is insufficient after the escalation threshold. Endpoint deprecation can mask as a transient vendor error, so periodic validation against the vendor's current documented URL is part of the diagnostic surface. Reference implementation: `ibkr_forecast_bot/scripts/flex_retry.py`; endpoint-drift instance: 2026-05-05 IBKR Flex legacy `/Universal/servlet/FlexStatementService.*` returned HTTP 404 while current Account Management Flex endpoint worked.
- Principle 27: Strategy naming must reflect actual mechanism, not initial framing or audit-time vocabulary. When a strategy's mechanism diverges from its name during development, correct the name explicitly in documentation, status reports, and operational vocabulary. Kill-switch paths and config keys can retain legacy names for operational continuity, but documentation must be unambiguous about actual mechanism. Current reference: Kalshi `deterministic_settlement_index` and `deterministic_settlement_fx` are live Index/FX Longshot Fade strategies, not proper cushion-based deterministic settlement; proper cushion Index/FX DS is a separate shadow-only research stream.
- Principle 28: Cap raises require post-raise sample evidence, not just lifetime evidence. A second or later cap lift should cite the cohort created by the prior lift, with resolved n, W-L, P&L, Wilson lower bound, and cap-binding evidence. Keep risk floors unchanged unless the explicit decision is a risk-budget change. Current references: Kalshi crypto DS max_open 6 -> 8 only after the 4 -> 6 cohort produced 98 resolved at 94W-4L with positive P&L and continued cap-binding; Kalshi crypto DS max_open 8 -> 10 only after the 6 -> 8 cohort produced 109+ resolved at 97%+ WR, Wilson lower above 90%, positive CB state, and sustained cap-binding.
- Principle 29: Shadow research methodology must validate at the cell level, not the aggregate level. Aggregate metrics on shadow research can hide structural data-quality issues that make the data not live-deployable. Per-cell decomposition (sport x contract_type x confidence_bucket; asset x contract_type x edge_bucket; city x cushion_direction x edge_bucket; etc.) is the validation standard. Decision-grade requires n>=50 per cell with Wilson lower bound above breakeven for that cell, not aggregate Wilson bounds across mixed cells. Counter-example: sports DS aggregate showed 66W-0L promising data, but per-cell decomposition revealed 0/619 properly-paired game-state outcomes due to NULL contract-type tags, completed-game pollution, future-contract mismatches, and multileg combo dominance; the rebuild restored cell-level validity at the cost of resetting accumulation. Application: every new shadow research stream must define its cell decomposition at design time, not retrofit it after accumulation. Reports must show per-cell metrics; aggregate metrics may be displayed but must be labeled as descriptive, not decision-grade.
- Principle 30: Production sizing changes require shadow-comparison evidence before activation. A new sizing model must first run beside the live sizing path, write insert-per-observation "would-have-sized" rows, and render in reports with actual-vs-shadow deltas. Activation is a separate operator decision after enough forward shadow-comparison rows show statistically meaningful improvement and after reconciliation, circuit breakers, kill switches, and strategy independence are verified. The 2026-05-03 Kelly hardening pass added Kalshi MR uncertainty/drawdown-aware Kelly, Kalshi DS clustered-settlement cohort tracking, and IBKR Long+Short Kelly as shadow/inactive infrastructure only; none changed live sizing.
- Principle 31: High-growth operational databases need lifecycle maintenance, not just ad hoc VACUUM. For append-heavy shadow stores, distinguish real data growth from reclaimable freelist/WAL growth before choosing maintenance. Online checkpoint/size monitoring should run continuously; full VACUUM/integrity windows should coordinate only non-live writers; partition/archive is the durable growth-control layer once data growth is real. The 2026-05-03 DS expansion DB maintenance found `freelist_count=0`, so repeated VACUUM will not solve main-file growth; the DS shadow DB now uses daily per-table resolution-aware archival into immutable monthly archives. Resolvers update active rows only, resolver-mutable rows stay active until safe, immutable observation rows can archive by age, and archive-aware readers are required for lifetime decision metrics.
- Principle 32: Operator-overridden live experiments require a pre-committed evidence boundary. If Eric explicitly overrides a pause/keep-shadow recommendation to get live data, treat the result as a new bounded pilot with its own cohort start, kill switch, tight CBs, low exposure, and explicit exit/expansion criteria before restart. Do not mix the prior failed strategy's CB ledger with the new pilot if the operator defines a materially different hypothesis; preserve the old rows for reporting, but scope the new cohort and CB ledger from the pilot start. Current reference: 2026-05-04 Gemini DS depth-filtered pilot tests placement-time depth >=500 across BTC/ETH/SOL with qty 25, max_open 2, daily CB -75, lifetime CB -150, and 50-resolution review criteria.
- Principle 33: Kelly-backed live sizing changes require conservative uncertainty adjustment plus forward cohort evidence. When Eric approves an aggressive sizing change from a sufficiently large live sample, anchor the deployed size to a defensible Kelly calculation that uses a conservative parameter estimate such as Wilson lower bound, scale CBs with the new loss magnitude, preserve strategy independence, and create a qty-specific forward cohort tracker before restart. Do not treat a qty cohort as evidence for max_open changes or vice versa. Current reference: 2026-05-04 Kalshi crypto DS qty 50 -> 225 used quarter-Kelly on Wilson 95% lower bound, retained max_open 10, scaled CBs to -500/-1500, and added `ds_qty_cohort_tracker.py`.
- Principle 34: Empirical analysis validates collection scope; it does not just validate strategies. Shadow categories with weeks of rows, no resolver/actionability path, and no decision-grade cells become collection kill/downsample candidates, but disable them with reversible platform-scoped config and monitor post-change write rates. Do not disable broad category names that would also stop productive per-cell streams. Current reference: 2026-05-05 DS infrastructure pass disabled only `kalshi:politics`, `kalshi:culture`, `kalshi:business`, `kalshi:tech`, and `kalshi:climate_science` after Phase 1-3 analysis plus fresh per-category verification; Gemini non-crypto and Gemini weather stayed enabled because fresh data showed actionable rows or per-cell weather coupling. When Eric decides that live execution data is more valuable than additional shadow data for a confirmed cell pattern, translate the cell surgically across venues, keep sizing small unless the venue-specific cohort already supports scale, instrument the causal fields needed for postmortem analysis, and start a new cohort from the deploy timestamp. Current reference: 2026-05-05 Kalshi/Gemini crypto DS high-price + high-cushion filters operationalize the same adverse-selection-resistant cell: price >=0.90 and cushion >=0.10%, with Gemini additionally requiring depth >=1000 at qty 10.
- Principle 36: High-growth hot stores must decouple write-path from active-residence. For append-heavy shadow databases, the active DB should be a hot operational store for current resolver lifecycle rows and recent decision data; future-analysis history belongs in archive tiers queried through archive-aware readers. Continuous micro-archive should move resolver-safe rows online using shared locks and small transactions, while maintenance-window sweeps handle VACUUM/ANALYZE compaction when `auto_vacuum=0`. Current reference: 2026-05-05 DS shadow continuous archive added `ds-shadow-continuous-archive.timer`, shared `data/ds_shadow_archive.lock`, and report-visible active direct stats so `ds_shadow_expansion.db` stays bounded between 6-hour compaction sweeps without deleting historical data.
- Principle 37: Parallel experimental design beats asymmetric folklore when two venues share a thesis. If two related strategies use different filters because one historical sample showed an extra variable, run a bounded parallel-filter pilot when operator-approved: match the shared structural filters, preserve strategy independence, start a fresh cohort boundary, and track the removed variable as an explanatory bucket instead of enforcing it. Current reference: 2026-05-05 Gemini DS removed depth enforcement to match Kalshi DS price>=0.90 plus cushion>=0.10%, while `gemini_ds_cohort_tracker.py` records per-depth-bucket outcomes to test whether depth adds explanatory power after price/cushion controls. This applies BUG_CLASSES #18: aggregate or uncontrolled historical slices can hide cell-level structural effects.
- Principle 38: Asymmetric-loss MR cells need hard cell-level exposure caps, not broad strategy tightening. When a high-WR strategy is near break-even because a narrow cell has catastrophic loss size, prefer a surgical hard reject on the cell's total cost or correlated cluster over resizing down or blocking unrelated profitable sides. Restricting concentrated loss sources requires restriction-grade evidence, not the higher expansion-grade bar used to scale profitable cells. The reject must live in both evaluation and pre-exec paths, log an explicit reason, create forward cohort evidence, and preserve strategy independence. Current references: 2026-05-05 Kalshi MR WEATHER_LOW YES cost >=$150 cap; 2026-05-07 Kalshi MR INDEX max 2 per underlier/event and max 4 per settlement-hour cluster cap. This applies BUG_CLASSES #18 and #21.
- Principle 39: Disable validated-empty broad collection while preserving productive per-cell streams. A high-growth category name can contain both wasteful broad universe rows and valuable strategy-specific rows, so disable at the narrow writer path or platform-scoped token only after fresh row/actionability/resolution checks prove the broad path is not decision-grade. Add per-category daily caps as a runaway-growth safety net, and verify post-change writes on both the disabled broad path and the preserved productive path. Current reference: 2026-05-05 DS shadow final DB fix disabled Gemini validated-empty broad categories, disabled sports broad backfill via `DS_SHADOW_SPORTS_BROAD_BACKFILL_ENABLED=false`, preserved `sports-ds-forward-scanner.service`, and added report-visible daily caps. This applies BUG_CLASSES #18 and #21.
- Principle 40: Side-aware aggregation is mandatory for side-specific prediction cells. If a detector can predict YES or NO, outcome aggregation must count a win only when `actual_outcome == predicted_side`; treating `actual_outcome='YES'` as a universal win inverts NO-side cells and can falsely kill profitable strategies. Deployment analyses must print the exact side-aware win expression or use a shared helper, decompose by side, and spot-check at least one NO win and one NO loss before any pilot decision. Current reference: 2026-05-05 H1 Kalshi crypto DS audit corrected BTC/ETH/SOL low-price NO cells from false kill/inconclusive classifications into pilot-worthy cells. This applies BUG_CLASSES #22.
- Principle 41: Single-asset concentration limits are permanent safety features, not deploy-time checks. If a sizing model can concentrate capital in one asset, the concentration cap must live in the future placement path and log the cap math on every affected decision; do not treat it as a one-time pre-restart calculation. Current reference: 2026-05-05 Gemini MR half-Kelly activation added a 50% per-asset concentration cap that blocks or sizes down new ETH placements while existing ETH exposure is above cap. This applies BUG_CLASSES #23.
- Principle 42: When safety constraints conflict with sizing preferences, safety wins. A Kelly or Markowitz/Thorp sizing model should compute from bankroll, edge, and hard risk constraints; do not add a "do not reduce current qty" floor when it contradicts concentration safety. Current reference: 2026-05-05 Gemini MR Kelly deployment removed the qty-floor operator preference after it conflicted with the permanent 50% per-asset concentration cap. This applies BUG_CLASSES #23.
- Principle 43: Near-settlement ladder strategies need correlated-event caps, not only per-symbol caps. Adjacent contracts in the same underlying and settlement hour are not independent; one fast underlying move can convert several high-confidence positions into simultaneous losses. When qty or capital changes make funded slots the practical limiter, verify selection order and add a placement-path asset/settlement-hour cap through the same sizing source-of-truth helper. Current reference: 2026-05-06 Kalshi crypto DS qty-400 review found all price/cushion filters and resolver accounting clean, but three losses came from one crypto settlement cluster; `DETERMINISTIC_SETTLEMENT_MAX_OPEN` was reduced 10 -> 5 and `DETERMINISTIC_SETTLEMENT_MAX_ASSET_HOUR_OPEN=4` was added inside `_ds_cap_available_qty()`.
- Principle 44: Audit-derived subset statistics must be re-derived from raw shadow data before deployment decisions. A processed audit slice is not decision-grade unless its exact source table, filter predicates, time window, side-aware win expression, and methodology class can be re-run against raw rows and reproduce the cited WR within 5 percentage points. If the re-derived raw population differs materially, treat the audit subset as over-fit or non-generalizable and either keep it shadow-only or define a narrower live pilot whose placement path preserves the exact subset criteria. Current references: 2026-05-04 H17 side-agnostic contamination caught before deploy; 2026-05-05 Index/FX wrong-table issue where cited high-WR data came from a different longshot methodology while the proper cushion table had n=0; 2026-05-06 Kalshi Low-Price NO review where the cited H1 90%+ subset did not match the current raw `shadow_deterministic_settlement` BTC/ETH/SOL NO <80c slice of 185W-88L (67.8%). This applies BUG_CLASSES #24.
- Principle 45: Scaling decisions in clustered binary markets must be asset-specific when P&L tails diverge. Similar WR across assets does not imply similar scale safety because settlement clustering, ladder density, and loss magnitude can differ by underlying. When a scaled cohort fails, decompose by asset before applying a blunt rollback; keep only assets whose scaled cohort meets P&L and CB discipline, roll back assets with structural tail loss, and start new per-asset forward cohorts. Current reference: 2026-05-06 Kalshi crypto DS qty-400 gate failed in aggregate at 27W-3L, Wilson lower 74.4%, P&L -$691.95, and daily CB breach, while BTC at qty 400 remained positive and ETH showed negative P&L despite similar WR. This applies BUG_CLASSES #25.
- Principle 46: Database cleanup must preserve operational evidence before optimizing size. Rows that support active cohorts, methodology re-derivation, paused-but-may-revive strategies, resolver correction windows, or audit provenance are not cleanup candidates even when large. Bounded cleanup is acceptable only for categories proven not applicable with zero lifetime actionable rows, zero resolved rows, and no active or paused strategy dependency; otherwise surface the option for operator decision. Current reference: 2026-05-06 DS shadow DB audit cleaned only active `kalshi|politics`, `kalshi|culture`, and `kalshi|business` signal rows after verifying zero actionable/resolved utility and preserved all active cohort and re-derivation data.
- Principle 47: Long-term archives should rotate to compressed recoverable tiers before deletion is considered. - Principle 51: Shadow surfaces report unit-of-independence, not raw row count. For any shadow scanner or research surface that records multiple observations per underlying contract, contract-date, or settlement event, the statistical confidence metrics (Wilson lower bound, Fisher exact, chi-square, t-test, gate evaluation) MUST use the unit of independent risk as the denominator, not the raw row count. Wilson lower bound at n=1000 with WR 100% suggests Wilson lower around 99.6%. That's statistically appropriate if n=1000 represents 1000 independent trade decisions. If those 1000 rows represent 10 contracts observed 100 times each, the true Wilson lower on the unit of independent risk is around 65%, not 99.6%. This methodology trap surfaced 2026-05-10 in the Gemini weather forensic where four LIVE_PILOT_CANDIDATE cells with row-level Wilson 93-99.8% collapsed to 4-7 independent executable contract-date groups, far below any gate-met threshold. The trap is most acute in shadow surfaces that scan periodically during contract lifetimes: weather cushion-DS minute-level NWS forecast snapshots, cushion-DS final-30m scanners, and similar architectures. Reports MUST show both row-level metrics (descriptive) and collapsed-by-unit-of-independence metrics (decision-grade). Row-level data is useful for descriptive analysis such as average cushion, observation density, and P&L attribution, but never for gate decisions. Gate criteria for live-pilot promotion, live-pilot evaluation, scale-up, or retire decisions MUST reference the collapsed unit, not row count. Pre-committed gates that currently reference row counts should be updated to the appropriate collapsed unit such as independent executable contract-date groups. New shadow surfaces must specify their unit of independence at design time. Default units: weather contracts use contract-date; cushion-DS final-30m scanners use contract-settlement; per-trade execution shadows such as depth filter and calibration semantics use per-trade rows because row count already equals unit count; Moderate Favorites and similar surfaces use per-contract or per-event depending on whether re-observation is captured.
